Guides d'Implémentation Avancés

Explorez les fonctionnalités opérationnelles profondes de Veridot V4, y compris le stockage des métadonnées, la révocation temps réel et la rotation des clés.

Révocation de Jetons

Veridot fournit deux niveaux de révocation : la révocation d'une session individuelle, et la révocation globale d'un groupe. La révocation s'effectue via l'interface TokenRevoker.

Aucune Suppression Physique

Dans la V4, révoquer ne supprime pas de lignes de la base de données. Le système écrit une enveloppe signée LIVENESS avec le statut REVOKED. Les vérificateurs locaux interceptent immédiatement ce signal et bloquent le jeton.

// Révoquer une session unique (via son sequenceId unique)
service.revoke(Scope.group("user-1234"), "session-xyz-123");

// Révoquer toutes les sessions d'un groupe (ex. déconnexion globale d'un utilisateur)
service.revoke(Scope.group("user-1234"), null);

Brokers de Métadonnées Distribués

Le broker est responsable de la réplication des messages. Veridot fournit des adaptateurs pour plusieurs backends :

1. Broker Apache Kafka (Recommandé pour la haute performance)

Assure une réplication temps réel ultra-rapide (inférieure à la seconde) à l'aide d'un log transactionnel distribué. Disponible dans veridot-kafka :

Properties kafkaProps = new Properties();
kafkaProps.put("bootstrap.servers", "localhost:9092");
kafkaProps.put("embedded.db.path", "/var/lib/veridot-rocksdb");
Broker kafkaBroker = new KafkaBroker(kafkaProps);

2. Broker Base de Données (SQL)

Utilise des bases relationnelles comme PostgreSQL, MySQL ou MS SQL Server comme broker de messages. Disponible dans veridot-databases :

HikariDataSource dataSource = new HikariDataSource();
dataSource.setJdbcUrl("jdbc:postgresql://localhost:5432/veridot");
Broker sqlBroker = new DatabaseBroker(dataSource, "broker_messages");

Rotation des Clés Éphémères

Afin de limiter les risques d'exposition des clés privées, Veridot effectue une rotation automatique en arrière-plan. Le KeyRotationService génère des paires de clés éphémères (RSA ou Ed25519) à courte durée de vie. Les clés publiques correspondantes sont publiées sous forme d'enveloppes KEY_EPOCH sur le broker afin que les vérificateurs puissent les récupérer et les valider dynamiquement.
