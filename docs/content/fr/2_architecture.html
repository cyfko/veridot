Architecture Système & Protocole V4

Veridot V4 fonctionne en publiant des entrées signées cryptographiquement vers un broker de métadonnées partagé à haute disponibilité. Les nœuds de vérification lisent ce broker et maintiennent une vue locale de l'état tout en garantissant des invariants de sécurité stricts.

Format de l'Enveloppe Binaire

Chaque message propagé par le broker est encapsulé dans une enveloppe binaire compacte (Envelope). Son format de sérialisation est structuré ainsi :

+-----------------+-------------------+-------------------+-------------------+
|  Magic (1B)     | Version (1B)      | Entry Type (1B)   | Flags (1B)        |
|  valeur : 0x04  | valeur : 0x01     | (ex. CONFIG)      | (bitmask)         |
+-----------------+-------------------+-------------------+-------------------+
|  Scope Length   | Scope Value (Var) | Key Length (2B)   | Key Value (Var)   |
+-----------------+-------------------+-------------------+-------------------+
|  Version (8B)   | Timestamp (8B)    | Issuer ID Length  | Issuer ID Value   |
+-----------------+-------------------+-------------------+-------------------+
|  Payload Length | Payload Bytes     | Signature Alg     | Signature Bytes   |
+-----------------+-------------------+-------------------+-------------------+

Types d'Entrées du Protocole

Le système utilise 5 types de messages principaux pour synchroniser l'état et appliquer les règles :

  • CONFIG : Définit la capacité maximale de sessions, les politiques d'éviction, les TTL par défaut et la validité de la configuration.
  • KEY_EPOCH : Publie les clés de rotation temporaires, les octets de clé publique éphémère et les signatures associées.
  • LIVENESS : Signaux de vie périodiques (heartbeats). Un état peut être ACTIVE (actif) ou REVOKED (révoqué).
  • FENCE : Établit l'ordre d'écriture et exclut (fence) les éditeurs concurrents ou obsolètes lors des modifications de capacité.
  • SNAPSHOT_MARKER : Publié périodiquement lors des réconciliations pour établir un nouveau filigrane anti-rollback.

Garanties Anti-Rollback & Réconciliation

Une vulnérabilité majeure des systèmes décentralisés est l'attaque par rejeu, où un attaquant réémet un ancien état "actif". Veridot élimine ce risque grâce à deux invariants clés :

Filigrane Strict (Watermarking)

Chaque nœud maintient un VersionWatermark qui suit la version de séquence la plus élevée vue pour chaque ressource. Si un nœud reçoit un message ayant une version inférieure ou égale au filigrane local, il est immédiatement rejeté.

Pour synchroniser les filigranes locaux, le ReconciliationManager exécute des boucles de réconciliation périodiques. Il effectue un snapshot du broker, agrège l'état actif, et publie un SNAPSHOT_MARKER signé pour aligner le filigrane de base sur toutes les instances en cours d'exécution.
