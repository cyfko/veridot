Sécurité Opérationnelle & Bonnes Pratiques

Exploiter Veridot dans un environnement de production exige le respect rigoureux des contraintes de synchronisation temporelle et de résolution des clés de confiance.

Synchronisation Temporelle & Dérive d'Horloge

Veridot s'appuie sur des timestamps absolus pour valider la durée de vie des sessions (via les entrées LIVENESS) et la validité des clés éphémères. Si les horloges des serveurs dérivent, la validation échouera par sécurité.

Seuil Strict de Dérive Temporelle (§9.1)

Le décalage d'horloge (clock skew) maximum toléré entre un signataire et un vérificateur est de 5 minutes. Les nœuds de production doivent synchroniser leurs horloges système via le protocole NTP pour maintenir une dérive inférieure à 50ms.

Configuration NTP Recommandée (exemple systemd-timesyncd)

Assurez-vous que timesyncd est actif et configuré dans /etc/systemd/timesyncd.conf :

[Time]
NTP=pool.ntp.org time.google.com
FallbackNTP=ubuntu.pool.ntp.org

Invariants de Sécurité du TrustRoot & KMS

L'interface TrustRoot résout les clés publiques nécessaires à la validation des enveloppes. Elle est le socle de sécurité de Veridot :

  • Conception Fail-Closed (Verrouillage par défaut) : Si le KMS ou le TrustRoot est indisponible, le vérificateur doit rejeter immédiatement tous les jetons plutôt que de tolérer des jetons non vérifiés.
  • Isolation des Émetteurs (Issuers) : Les clés éphémères doivent être validées avec la clé à long terme de l'émetteur concerné. La compromission d'un émetteur n'affecte pas les autres, garantissant l'étanchéité multi-tenant.

Référence des Codes d'Erreur du Protocole

Lorsque le protocole rencontre un problème, les exceptions contiennent des codes d'erreur standardisés. Référez-vous à ce tableau pour le diagnostic :

Code Identifiant Description / Résolution
0x01 INVALID_SIGNATURE La signature de l'enveloppe est corrompue ou signée par une clé non autorisée par le TrustRoot.
0x02 EXPIRED_LIVENESS La session a dépassé sa durée de vie heartbeat. L'utilisateur a été déconnecté ou le thread renouvellement est arrêté.
0x03 FENCE_TOKEN_STALE Conflit d'écriture concurrent. Un signataire plus récent a pris le contrôle de la capacité. Réessayez l'opération.
0x04 CAPACITY_EXCEEDED Le nombre maximal de sessions concurrentes défini dans CONFIG est atteint et la politique d'éviction est REJECT.
0x05 TRANSPORT_UNAVAILABLE Le vérificateur ne peut plus lire/écrire sur le broker de métadonnées (ex. partition Kafka hors-ligne ou base SQL injoignable).
