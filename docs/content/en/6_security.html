Operational Security & Constraints

Using Veridot in a secure manner requires understanding and maintaining key environment requirements, specifically regarding clock synchronization and cache configuration.

Clock Synchronization & Skew

Because Veridot signs and validates liveness heartbeats containing absolute UNIX timestamps, system clocks on all nodes must be synchronized using NTP (Network Time Protocol).

Maximum Allowed Clock Skew

The maximum tolerated clock skew between any signer and any verifier is 5 minutes. If a verifier detects a token timestamp skewing beyond this threshold, validation will fail-safe and reject the token.

Authorization Capability Caching

To keep latency extremely low (sub-millisecond), verifying nodes cache publisher authorization capabilities locally using TTLs. However, this introduces a latency window during which a revoked capability might still be treated as valid.

Positive Caching (Default: 60s)

Controls how long a verified publisher capability is trusted. A lower TTL reduces revocation detection delay but increases network lookup overhead on the broker.

Negative Caching (Default: 5s)

Controls how long a failed authorization check is cached. This is critical to prevent denial-of-service (DoS) patterns where an unauthorized node bombards the verifier with requests, triggering database lookup loops.

Recommended Algorithms

Use modern, strong cryptographic algorithms to protect the envelope signatures:

  • Ed25519 (Recommended): Provides high-speed signature verification and small signature sizes, ideal for high-throughput API gateways.
  • RSA 3072/4096: Secure choice for compatibility with enterprise HSMs. Avoid RSA keys smaller than 2048 bits.
