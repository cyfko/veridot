Operational Security & Guidelines

Operating Veridot in a secure environment requires strict adherence to system requirements, clock synchronization, and proper key hierarchy resolution.

Clock Synchronization & Skew Constraints

Veridot relies on absolute time thresholds to establish session liveness (via LIVENESS entries) and key epoch validity. If system clocks drift significantly across instances, validation failures or window bypasses might occur.

Strict Time Skew Threshold (§9.1)

The maximum allowed clock skew between any signer node and any verifying gateway is 5 minutes. Gateways must synchronize their system clocks using NTP (Network Time Protocol) to ensure skews remain well below 50 milliseconds in production.

Recommended NTP Configuration (systemd-timesyncd example)

Ensure timesyncd is active and configured with reliable pools in /etc/systemd/timesyncd.conf:

[Time]
NTP=pool.ntp.org time.google.com
FallbackNTP=ubuntu.pool.ntp.org

TrustRoot & KMS Security Invariants

The TrustRoot interface resolves public keys for validating the signatures on envelopes. It forms the foundation of Veridot's security model:

  • Fail-Safe Design: If the TrustRoot becomes unavailable (e.g. KMS connection lost), the verifier must immediately fail-closed and reject all incoming tokens rather than falling back to trust-all mode.
  • Issuer Isolation: Ephemeral keys must be signed by the corresponding long-term issuer key. A compromise of an issuer key only affects the sessions created by that specific issuer, preventing lateral tenant compromise.

Protocol Error Codes Reference

When validation or operation fails, Veridot throws exceptions containing standardized error codes. Use this table to diagnose issues:

Error Code Identifier Description / Resolution
0x01 INVALID_SIGNATURE The envelope signature is corrupt or signed by an untrusted key. Check TrustRoot configuration.
0x02 EXPIRED_LIVENESS The session has exceeded its heartbeat TTL. The user is logged out or the signer renewal thread stopped.
0x03 FENCE_TOKEN_STALE A concurrency write conflict occurred. A newer signer node has taken ownership. Retry the operation.
0x04 CAPACITY_EXCEEDED The maximum concurrent session capacity defined in CONFIG has been reached and policy is set to REJECT.
0x05 TRANSPORT_UNAVAILABLE The verifier cannot read or write to the shared metadata broker (ex. Kafka partition offline, SQL database down).
