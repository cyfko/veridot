Introduction to Veridot

Veridot is an enterprise-grade, high-performance cryptographic token verification protocol and Java library designed for large-scale distributed architectures. It bridges the gap between fast, stateless local JWT verification and instant, stateful token revocation.

The Distributed Revocation Paradox

Modern microservice architectures require fast, secure, and highly scalable user authentication. In typical setups, developers choose between two flawed architectural patterns:

The Authentication Trade-Off

1. Stateful Token Introspection (OAuth2 / Database Lookup)
Verifying services perform synchronous network queries to a central authorization server or database for every API request. While this allows immediate token revocation, it creates a severe performance bottleneck, adds 10-100ms of latency per request, and creates a Single Point of Failure (SPOF).

2. Stateless JWT Validation
Verifying services validate JSON Web Tokens (JWT) locally using a cached public key. This is extremely fast (sub-millisecond) and highly resilient to network partitions. However, once a token is signed, it cannot be revoked until its expiration time (TTL) has elapsed, exposing the system to security vulnerabilities if a token is compromised.

The Veridot Solution

Veridot resolves this paradox. It provides sub-millisecond local verification latency while maintaining near-instant revocation guarantees across multiple datacenters. It achieves this through a replicated metadata architecture:

  • Local Cryptographic Cache: Verifiers maintain a local, thread-safe cache of active session keys, public rotation keys, and configuration constraints.
  • Asynchronous Status Replication: State transitions (new sessions, revocations, policy changes) are published to a distributed high-throughput log (like Apache Kafka or relational databases) and propagated asynchronously to all verifiers.
  • Version Watermark Invariants: A strict mathematical version watermarking protocol protects the system against replay attacks, network partitions, and stale snapshot rollbacks.

Real-World Scenarios Solved by Veridot

Scenario A: The Compromised API Key

An attacker steals a user's API key. Under standard JWT, the key remains valid until its 1-hour expiration time. With Veridot, the security team triggers a revocation event. Within milliseconds, the revocation is propagated to all API Gateways, and subsequent requests using that token are rejected.

Scenario B: Central Database Outage

In a stateful introspection setup, if the authentication database crashes, all downstream microservices stop functioning. Under Veridot, verifiers continue validating tokens using their local cache and the last-known trust state, allowing normal operation during the outage.
