Advanced Implementation Guides

This section details how to integrate Veridot into production backends, manage cache thresholds, and deploy distributed brokers.

Production Database Broker (SQL)

Using a relational database (PostgreSQL, MySQL, SQL Server) as a broker is ideal for standard microservices. It requires a connection pool (such as HikariCP) and a dedicated table to store the binary envelopes. Here is a production-ready setup:

import com.zaxxer.hikari.HikariConfig;
import com.zaxxer.hikari.HikariDataSource;
import io.github.cyfko.veridot.core.Broker;
import io.github.cyfko.veridot.databases.DatabaseBroker;

public class DatabaseBrokerFactory {
    public static Broker createPostgresBroker() {
        HikariConfig config = new HikariConfig();
        config.setJdbcUrl("jdbc:postgresql://db-server.internal:5432/veridot_db");
        config.setUsername("veridot_app");
        config.setPassword("secure-database-password");
        
        // Performance optimization properties
        config.setMaximumPoolSize(15);
        config.setMinimumIdle(5);
        config.setIdleTimeout(30000);
        config.setConnectionTimeout(5000);
        config.addDataSourceProperty("cachePrepStmts", "true");
        config.addDataSourceProperty("prepStmtCacheSize", "250");
        config.addDataSourceProperty("prepStmtCacheSqlLimit", "2048");

        HikariDataSource dataSource = new HikariDataSource(config);
        
        // "broker_messages" is the database table where entries are stored
        return new DatabaseBroker(dataSource, "broker_messages");
    }
}

Database Schema (PostgreSQL DDL example)

CREATE TABLE broker_messages (
    storage_key BYTEA PRIMARY KEY,
    entry_bytes BYTEA NOT NULL,
    version BIGINT NOT NULL,
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_broker_messages_version ON broker_messages(version);

Production Apache Kafka Broker

For large-scale, multi-region deployments, Apache Kafka is the recommended metadata broker. It provides sub-second event replication and scales horizontally. The KafkaBroker uses an embedded **RocksDB** instance for lightning-fast local cache reads:

import io.github.cyfko.veridot.core.Broker;
import io.github.cyfko.veridot.kafka.KafkaBroker;
import java.util.Properties;

public class KafkaBrokerFactory {
    public static Broker createKafkaBroker() {
        Properties props = new Properties();
        
        // Kafka cluster servers
        props.put("bootstrap.servers", "kafka-broker-01:9092,kafka-broker-02:9092");
        
        // Embedded RocksDB persistent cache directory
        props.put("embedded.db.path", "/var/lib/veridot/rocksdb_cache");
        
        // Kafka topic for replication (must be pre-created with replication factor >= 3)
        props.put("broker.topic", "veridot-state-replication");
        
        // Producer / Consumer security configs
        props.put("security.protocol", "SASL_SSL");
        props.put("sasl.mechanism", "PLAIN");
        props.put("sasl.jaas.config", "org.apache.kafka.common.security.plain.PlainLoginModule required username=\"app\" password=\"secret\";");

        return new KafkaBroker(props);
    }
}

Protecting Databases via Negative Caching

In distributed systems, API gateways are often bombarded with unauthorized or malformed requests. If every failed authorization check triggers a synchronous query to the database, the database can easily get overloaded (a Denial of Service vulnerability).

How Negative Caching works

When a verifier checks a token signed by an unauthorized publisher, the authorization check fails. Veridot caches this failure locally for a short duration (configured via VDOT_CAPABILITY_NEGATIVE_CACHE_TTL_SECONDS, default 5s). During this window, all subsequent requests from that publisher are immediately rejected locally without querying the database broker, shielding your database from high-throughput brute-force attacks.
