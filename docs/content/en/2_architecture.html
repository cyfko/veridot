System Architecture & Protocol V4

Veridot V4 operates by publishing cryptographically signed entries to a shared, high-availability metadata broker. Verifying nodes read from this broker, maintaining a local view of the state while guaranteeing safety invariants.

Binary Envelope Format

Every message propagated through the broker is wrapped in a compact binary Envelope. The wire format is structured as follows:

+-----------------+-------------------+-------------------+-------------------+
|  Magic (1B)     | Version (1B)      | Entry Type (1B)   | Flags (1B)        |
|  value: 0x04    | value: 0x01       | (e.g. CONFIG)     | (bitmask)         |
+-----------------+-------------------+-------------------+-------------------+
|  Scope Length   | Scope Value (Var) | Key Length (2B)   | Key Value (Var)   |
+-----------------+-------------------+-------------------+-------------------+
|  Version (8B)   | Timestamp (8B)    | Issuer ID Length  | Issuer ID Value   |
+-----------------+-------------------+-------------------+-------------------+
|  Payload Length | Payload Bytes     | Signature Alg     | Signature Bytes   |
+-----------------+-------------------+-------------------+-------------------+

Protocol Entry Types

The system uses 5 primary message types to synchronize states and enforce rules:

  • CONFIG: Defines maximum session capacities, eviction policies, default TTLs, and configuration validity.
  • KEY_EPOCH: Publishes rotation keys, ephemeral public key bytes, validity intervals, and signatures.
  • LIVENESS: Periodic heartbeats sent by active signers. A state can be marked as ACTIVE or REVOKED.
  • FENCE: Used to establish write order and fence out stale or concurrent publishers during state mutations.
  • SNAPSHOT_MARKER: Published periodically during reconciliation to establish a fresh anti-rollback watermark.

Anti-Rollback & Reconciliation

A major vulnerability in decentralized status systems is the replay attack, where an attacker intercepts a stale "active" status and republishes it. Veridot eliminates this vulnerability using two invariants:

Strict Watermarking

Every node maintains a VersionWatermark that tracks the highest seen sequence version for every resource scope. If a node receives a message with a version number less than or equal to the local watermark, it is immediately discarded as stale.

To reconcile local watermarks with the broker state, the ReconciliationManager runs periodic reconciliation loops. It snapshots the broker, aggregates active states, and publishes a signed SNAPSHOT_MARKER to align the watermark baseline across all running instances.
