Core Concepts

This page describes the key concepts and cryptographic structures that make up the Veridot V4 protocol.

1. Binary Envelope Wire Format (§3)

Every piece of data published to the replication broker uses a single, unified binary envelope layout. This format ensures that signatures cannot be relocated and that message parsing requires zero-copy allocations:

Byte Offset | Size (Bytes) | Field Name         | Description
------------+--------------+--------------------+-----------------------------------
0           | 2            | Magic Bytes        | Must be 0x56 0x44 ("VD")
2           | 1            | Proto Version      | Must be 0x04
3           | 1            | Entry Type         | Type of message payload (0x01 - 0x06)
4           | 1            | Flags              | Bit 0: COMPACT_SIG (1=Ed25519, 0=RSA)
5-6         | 2            | Scope Length       | Length of the resource Scope string
7-X         | Var          | Scope Value        | UTF-8 string (ex. "group:user-123")
X+1 - X+2   | 2            | Key Length         | Length of the sub-key identifier
X+3 - Y     | Var          | Key Value          | UTF-8 string (ex. "session-9942")
Y+1 - Y+8   | 8            | Version            | Monotonically increasing sequence version
Y+9 - Y+16  | 8            | Timestamp          | Epoch milliseconds of publication
Y+17 - Y+18 | 2            | Issuer ID Length   | Length of publisher identifier
Y+19 - Z    | Var          | Issuer ID Value    | UTF-8 string of signer identity
Z+1 - Z+4   | 4            | Payload Length     | Length of the inner payload
Z+5 - W     | Var          | Payload Bytes      | Encoded inner payload bytes
W+1 - W+2   | 2            | Signature Algorithm| Signature alg (0x01=RSA-SHA256, 0x02=Ed25519)
W+3 - W+4   | 2            | Signature Length   | Length of the signature
W+5 - End   | Var          | Signature Bytes    | Signature over canonical signing bytes

2. Key Epoch (§5)

A KEY_EPOCH entry distributes ephemeral cryptographic keys used to verify tokens. Signers rotate keys periodically to minimize key exposure. A key epoch defines the temporal validity window:

  • validFrom: Timestamp from which the public key is valid (with 5-minute clock skew tolerance).
  • validUntil: Timestamp after which the key is expired and must not be used for verification.

3. Fencing and FENCE Entries (§9)

To support capacity constraints (like a limit of 3 concurrent active sessions per user account), Veridot uses FENCE entries to order mutations across concurrent signers without using heavy distributed locks.

A FENCE entry is a singleton per scope that contains a strictly increasing fenceCounter. A signer must publish a new FENCE entry to the broker before committing any capacity change. If another signer writes a higher fence counter, the current signer's counter becomes stale, and its write operations are rejected by the broker.

4. Version Watermark & Monotonicity (§11)

To guard against replay attacks and database rollback issues (where an old database backup is loaded, resetting revoked tokens to active), Veridot enforces the Monotonic Version Invariant.

The Monotonicity Rule

For every unique entry identified by its EntryId (scope, entryType, key), verifiers maintain a local watermark version. When an update arrives:

If the update's version is less than or equal to the local watermark, it is discarded immediately.

If the update's version is greater than the local watermark, the state is updated and the local watermark is set to the new version.
