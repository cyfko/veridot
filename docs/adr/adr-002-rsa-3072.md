# ADR-002 — RSA-3072 pour les clés éphémères

**Statut** : Accepté  
**Date** : 2026-06-27  
**Auteurs** : Frank KOSSI (Kunrin SA)  
**Implémenté dans** : `veridot-core` v3.0.2

---

## Contexte

Veridot utilise des paires de clés RSA **éphémères** pour signer les JWT. Jusqu'en v3.0.1, la taille de clé était **implicitement RSA-2048**, dérivée du comportement par défaut du JDK `KeyPairGenerator` sans argument `initialize()`.

Ce choix posait deux problèmes :

1. **Opacité** : aucune constante publique ne documentait la taille de clé effective. Un opérateur ne pouvait pas savoir quelle taille de clé son déploiement utilisait sans lire le bytecode.
2. **Obsolescence annoncée** : NIST SP 800-57 Part 1 Rev. 5 prévoit la dépréciation de RSA-2048 entre 2030 et 2035 pour les nouvelles applications. Migrer de manière proactive évite un second breaking change imposé dans 4–8 ans.

---

## Décision

Utiliser **RSA-3072** explicitement, avec la constante publique `Config.ASYMMETRIC_KEY_SIZE = 3072`, documentée et référencée partout où une clé éphémère est générée.

```java
// Config.java
/**
 * RSA key size (bits) for ephemeral signing key pairs.
 * RSA-3072 provides 128-bit security equivalent (NIST SP 800-57, Table 2).
 * This exceeds RSA-2048 (112-bit) and satisfies NIST recommendations through 2030+.
 */
public static final int ASYMMETRIC_KEY_SIZE = 3072;

// GenericSignerVerifier.java — usage
KeyPairGenerator generator = KeyPairGenerator.getInstance(Config.ASYMMETRIC_KEYPAIR_ALGORITHM);
generator.initialize(Config.ASYMMETRIC_KEY_SIZE, new SecureRandom());
KeyPair ephemeralKeyPair = generator.generateKeyPair();
```

## Analyse comparative

### Niveau de sécurité et performance

| Taille RSA | Sécurité équivalente | Référence | Perf. relative (sign) | Recommandation NIST |
|------------|---------------------|-----------|----------------------|---------------------|
| RSA-2048 | 112 bits | AES-112 | 1× (référence) | Acceptable jusqu'à 2030 |
| **RSA-3072** | **128 bits** | **AES-128, ECDSA P-256** | **~2×** | **Recommandé 2031+** |
| RSA-4096 | ~140 bits | AES-140 | ~4× | Sur-dimensionné pour clés éphémères |

RSA-3072 atteint 128 bits de sécurité équivalente — le même niveau que AES-128 et ECDSA P-256 — sans l'overhead prohibitif de RSA-4096. Pour des clés **éphémères rotées toutes les 24h**, ce compromis performance/sécurité est optimal.

### Pourquoi pas ECDSA P-256 à la place ?

ECDSA P-256 offrirait les mêmes 128 bits avec des clés et signatures 10× plus courtes. Ce changement est sur la **roadmap** (voir section Future), mais nécessiterait une migration du format des messages V2 (le champ `pubkey` stocke actuellement un DER RSA). RSA-3072 est le chemin de moindre résistance pour v3.0.2.

---

## Compatibilité

- Les tokens déjà émis avec des clés RSA-2048 restent **vérifiables jusqu'à leur TTL** : la taille de clé est encodée dans les octets DER de la clé publique stockée dans les métadonnées V2, pas dans un champ explicite séparé.
- Il n'y a **aucune migration de données** à effectuer — les anciens tokens expirent naturellement.
- La transition est **transparente** pour les consommateurs : ils continuent de lire le champ `pubkey` et de reconstruire la `PublicKey` depuis les octets DER, quelle que soit la taille.

---

## Conséquences

### Positives

- Niveau de sécurité documenté et aligné avec les recommandations NIST actuelles.
- Constante publique `Config.ASYMMETRIC_KEY_SIZE` : les audits de sécurité peuvent vérifier la taille de clé sans inspecter le bytecode.
- Préparation anticipée pour la dépréciation NIST 2030–2035 de RSA-2048.
- Aucune migration de données.

### Négatives / Coût

- Génération de clé éphémère ~2× plus lente. Négligeable pour une rotation toutes les 24h : la génération représente une opération unique au démarrage ou à la rotation.
- Signature JWT ~2× plus lente (~0.5–1ms sur un CPU moderne). Reste dans le SLA de `sign()` pour la grande majorité des déploiements.
- Messages V2 légèrement plus grands (clé publique RSA-3072 DER ≈ 422 octets vs 294 octets pour RSA-2048, avant base64url).

### Futur : migration ECDSA

La roadmap prévoit d'introduire ECDSA P-384 comme algorithme alternatif pour les clés éphémères, offrant 192 bits de sécurité avec des clés et signatures significativement plus courtes. Cette migration nécessitera une version du protocole V3.
