Operational Security & Guidelines

Operating Veridot in a production environment requires strict adherence to system requirements, clock synchronization, and key custody practices.

Clock Synchronization (NTP)

Veridot relies on absolute timestamps to establish key validity windows and session heartbeats. System clock drift across nodes can cause validation bypasses or unexpected rejections.

Strict Time Skew Threshold

The maximum allowed clock skew between any signer node and any verifying gateway is 5 minutes. All nodes must synchronize their system clocks using NTP (Network Time Protocol).

NTP Configuration Example (systemd-timesyncd)

Configure timesyncd on your Linux instances by editing /etc/systemd/timesyncd.conf:

[Time]
NTP=pool.ntp.org time.google.com
FallbackNTP=ubuntu.pool.ntp.org

Activate and verify synchronization status using the terminal:

sudo systemctl restart systemd-timesyncd
timedatectl status

KMS & TrustRoot Invariants

The TrustRoot interface is the root of trust for all cryptographic validations. Ensure your implementation follows these security invariants:

  • Fail-Closed Design: If your TrustRoot fails to reach the KMS, it must throw an exception and block verification. It must never fall back to accepting unverified tokens.
  • Least Privilege: Verifier nodes only require read access to the broker and public key resolution. Private keys must be restricted to signers and isolated inside the KMS.
  • Key Rotation: Rotate long-term TrustRoot keys annually. Ephemeral epoch keys should be rotated hourly (configured via VDOT_KEYS_ROTATION_MINUTES).
