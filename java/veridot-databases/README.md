# veridot-databases

[![Maven Central](https://img.shields.io/maven-central/v/io.github.cyfko/veridot-databases.svg)](https://central.sonatype.com/artifact/io.github.cyfko/veridot-databases)
[![License: MIT](https://img.shields.io/badge/License-MIT-blue.svg)](https://opensource.org/licenses/MIT)

SQL database-backed implementation of the [`MetadataBroker`](https://github.com/cyfko/veridot/blob/main/java/veridot-core/src/main/java/io/github/cyfko/veridot/core/MetadataBroker.java) interface for **Veridot**, enabling distributed token verification metadata propagation via relational databases.

---

## ✨ Features

- 🔐 **JWT Signing & Verification** using ephemeral RSA key pairs
- 🔁 **Automatic Key Rotation** (configurable interval)
- 📬 **Public key distribution** via SQL database (supports clustering and replication)
- ✅ **Pluggable table name** (safe from SQL injection)
- ✅ **Auto-create** message table on startup
- ✅ **Async operations** using `CompletableFuture`
- ⚙️ **Works with major databases**: PostgreSQL, MySQL, MariaDB, SQL Server, H2

## 📦 Table Structure

The broker automatically creates the following table if it does not exist:

```sql
CREATE TABLE broker_messages (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    message_key VARCHAR(255) NOT NULL UNIQUE,
    message_value TEXT NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

## 🚀 Usage

### Creating the broker

```java
import io.github.cyfko.veridot.databases.DatabaseMetadataBroker;
import io.github.cyfko.veridot.core.MetadataBroker;
import javax.sql.DataSource;

DataSource dataSource = createDataSource(); // Your JDBC DataSource

// Default table name "broker_messages"
MetadataBroker broker = new DatabaseMetadataBroker(dataSource);

// Custom table name
MetadataBroker broker = new DatabaseMetadataBroker(dataSource, "my_veridot_messages");
```

### Sign, verify, and revoke

```java
import io.github.cyfko.veridot.core.*;
import io.github.cyfko.veridot.core.impl.*;

// Create signer/verifier with the database broker
var sv = new GenericSignerVerifier(broker, "my-secret-salt");

// Sign data — returns JWT (DIRECT mode)
String jwt = sv.sign("john@example.com",
    BasicConfigurer.builder()
        .groupId("user-123")
        .validity(300)  // 5 minutes
        .build());

// Verify from any service connected to the same database
String email = sv.verify(jwt, String::toString);

// Revoke
sv.revoke(jwt);

// Verify after revocation → throws BrokerExtractionException
sv.verify(jwt, String::toString); // throws!
```

### With session management

```java
// Max 3 concurrent sessions per user, FIFO eviction
var sv = new GenericSignerVerifier(broker, "salt", 3, GenericSignerVerifier.EvictionPolicy.FIFO);

// Track active sessions
boolean hasActive = sv.hasActiveToken("user-123");

// Revoke all sessions for a user
sv.revokeGroup("user-123");
```

### INDIRECT mode (store token in broker)

```java
// Token stored in database, caller receives only the messageId
String messageId = sv.sign("sensitive-data",
    BasicConfigurer.builder()
        .groupId("api-service")
        .distribution(DistributionMode.INDIRECT)
        .validity(3600)
        .build());
// messageId = "2:api-service:<uuid>"

// Verify using messageId
String data = sv.verify(messageId, String::toString);
```

## 🏗️ Cluster / Distributed Considerations

- All nodes/services should point to the **same database** (or synchronized replicas)
- `message_key` column with `UNIQUE` constraint ensures consistency without global coordination
- If you use database-level replication, `DatabaseMetadataBroker` naturally benefits from high availability

## 🔐 Security Considerations

- **Table name validation**: Only alphanumeric characters and underscores are allowed (prevents SQL injection)
- **Auto-create**: The broker inspects database metadata and creates the table if missing — no manual setup required
- Ephemeral RSA key pairs with configurable rotation
- ±5 minute clock drift validation (Protocol V2 §9.1)
