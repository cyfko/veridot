Concepts Clés

Cette page détaille les structures cryptographiques et les principes de conception du protocole Veridot V4.

1. Format de l'Enveloppe Binaire Wire (§3)

Chaque message publié sur le broker utilise un format binaire strict et auto-décrit pour empêcher le déplacement de signatures et permettre un décodage rapide :

Offset (Octet) | Taille (Octets) | Nom du Champ       | Description
---------------+-----------------+--------------------+-----------------------------------
0              | 2               | Magic Bytes        | Doit être 0x56 0x44 ("VD")
2              | 1               | Proto Version      | Doit être 0x04
3              | 1               | Entry Type         | Type de message (0x01 - 0x06)
4              | 1               | Flags              | Bit 0: COMPACT_SIG (1=Ed25519, 0=RSA)
5-6            | 2               | Scope Length       | Longueur de la chaîne de Scope
7-X            | Var             | Scope Value        | Chaîne UTF-8 (ex. "group:user-123")
X+1 - X+2      | 2               | Key Length         | Longueur de la sous-clé
X+3 - Y        | Var             | Key Value          | Chaîne UTF-8 (ex. "session-9942")
Y+1 - Y+8      | 8               | Version            | Version incrémentale monotone
Y+9 - Y+16     | 8               | Timestamp          | Temps de publication (millisecondes)
Y+17 - Y+18    | 2               | Issuer ID Length   | Longueur de l'identifiant de l'émetteur
Y+19 - Z       | Var             | Issuer ID Value    | Identifiant de l'autorité émettrice
Z+1 - Z+4      | 4               | Payload Length     | Longueur du payload interne
Z+5 - W        | Var             | Payload Bytes      | Octets encodés du payload interne
W+1 - W+2      | 2               | Signature Algorithm| Algorithme (0x01=RSA-SHA256, 0x02=Ed25519)
W+3 - W+4      | 2               | Signature Length   | Longueur de la signature
W+5 - End      | Var             | Signature Bytes    | Signature cryptographique

2. Key Epoch (§5)

Une entrée KEY_EPOCH distribue les clés publiques éphémères servant à valider les signatures des jetons. La validité est encadrée par deux marqueurs temporels :

  • validFrom : Début de validité de la clé (incluant une dérive d'horloge de 5 minutes).
  • validUntil : Fin de validité, après laquelle la clé ne peut plus être acceptée.

3. Fencing et jetons FENCE (§9)

Pour gérer les limites de session concurrentes sans recourir à des verrous distribués lourds, Veridot utilise des entrées FENCE.

Chaque changement de capacité doit être précédé de l'écriture d'une entrée FENCE contenant un compteur fenceCounter incrémental. Si un autre serveur écrit un compteur plus élevé, le jeton du serveur actuel est considéré comme obsolète et rejeté, annulant la transaction.

4. Invariant de Monotonicité et Watermark (§11)

Pour contrer les attaques par rejeu de messages d'authentification obsolètes ou les restaurations accidentelles de bases de données, Veridot impose un invariant de version monotone.

Règle de Monotonicité

Pour chaque EntryId (scope, entryType, key), les validateurs enregistrent la dernière version acceptée :

Si la version reçue est inférieure ou égale au filigrane (watermark) local, le message est rejeté.

Si la version reçue est strictement supérieure, l'état local est mis à jour et le filigrane prend la valeur de cette version.
