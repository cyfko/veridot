System Architecture & Protocol V4

Veridot V4 is built on a highly optimized, event-driven replication model. The protocol ensures that all verifying nodes maintain security state consistency without central coordination bottlenecks.

Binary Envelope Wire Format

All state transitions (e.g., CONFIG, KEY_EPOCH, LIVENESS) are packed into a cryptographically signed binary envelope before replication. The exact layout is designed for fast, zero-copy parsing:

Byte Offset | Size (Bytes) | Field Name         | Description
------------+--------------+--------------------+-----------------------------------
0           | 1            | Magic Version      | Protocol identifier. Must be 0x04
1           | 1            | Entry Type         | Type of message payload (1-5)
2           | 1            | Flags              | Bitmask for compression or state
3           | 1            | Sig Alg            | Signature algorithm identifier (ex. 0x01)
4-5         | 2            | Scope Length       | Length of the resource Scope string
6-X         | Var          | Scope Value        | UTF-8 string (ex. "group:user-123")
X+1 - X+2   | 2            | Key Length         | Length of the sub-key identifier
X+3 - Y     | Var          | Key Value          | UTF-8 string (ex. "session-9942")
Y+1 - Y+8   | 8            | Version            | Monotonically increasing sequence version
Y+9 - Y+16  | 8            | Timestamp          | Epoch milliseconds of publication
Y+17 - Y+18 | 2            | Issuer ID Length   | Length of publisher identifier
Y+19 - Z    | Var          | Issuer ID Value    | UTF-8 string of signer identity
Z+1 - Z+4   | 4            | Payload Length     | Length of the inner payload
Z+5 - W     | Var          | Payload Bytes      | Encoded inner payload bytes
W+1 - W+2   | 2            | Signature Length   | Length of the cryptographic signature
W+3 - End   | Var          | Signature Bytes    | Signature over canonical signing bytes

Entry Payload Types

Veridot defines 5 specialized payload structures encoded using protobuf or compact TLV:

  • CONFIG (Type 0x01): Dictates group capacity limits (e.g., maximum 5 active devices), eviction strategies (FIFO / REJECT), and default TTL windows.
  • KEY_EPOCH (Type 0x02): Represents ephemeral keys rotated by signers. Contains the raw public key bytes, rotation validity range, and the optional encrypted payload JWT (in indirect mode).
  • LIVENESS (Type 0x03): Heartbeat signal of active sessions. Status is set to ACTIVE (regular updates) or REVOKED (explicit logout).
  • FENCE (Type 0x04): Fencing tokens containing a monotonic fence counter. Used to validate state modifications and isolate concurrent publishers.
  • SNAPSHOT_MARKER (Type 0x05): Published periodically during global database reconciliation. Sets a baseline watermark across all verifiers.

Version Watermarking & Anti-Rollback

To prevent replay attacks (where an attacker replays a stale LIVENESS ACTIVE envelope to reactivate a revoked session), Veridot enforces the Monotonic Version Invariant:

Monotonicity Rule

For every unique resource identifier (Scope + Entry Type + Key), verifiers maintain a local watermark W(id). When a new envelope is received with version V:

If V ≤ W(id), the envelope is rejected.

If V > W(id), the envelope is accepted and W(id) ← V.

This ensures that status updates can never be rolled back to a previous state, even if the database is restored from a backup or suffers from replica lag.
