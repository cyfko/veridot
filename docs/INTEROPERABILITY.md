Interoperability
================

To ensure compatibility across programming languages, **Veridot** standardizes the events exchanged through a broker. That event should be made of a `key` and a `message` with the following format:  

- **Key:** The broker's event key is a string that represents a unique identifier. This guarantees uniqueness and seamless integration across diverse systems.
- **Message:** The broker's event message is a string that conform to the following structure:  
  - *[token config]*`:`*[Base64 RSA public key]*`:`*[Expiry date seconds]*`:`*[Base64 variant]*

  **Components:**  
  - **[token config]:** Configuration details for the token. Current implementations accept either `jwt` or `id` as valid values.  
  - **[Base64 RSA public key]:** Encoded RSA public key, used for verifying tokens.  
  - **[Expiry date seconds]:** Specifies the expiration time in seconds, enabling automatic removal of outdated entries.  
  - **[Base64 variant]:** Represents additional data related to the `[token config]`.  
    1. For the `jwt` *token config,* this value is optional and may be omitted.  
    2. For the `id` *token config,* this value contains the JWT to be verified or extracted using the public key.

## 🔐 Protocol V2 Message ID Format

**Veridot** uses **structured protocol identifiers** as broker event keys. Message IDs are no longer generated by hashing; they follow a deterministic, human-readable format defined by Protocol V2.

---

### ✨ Goals

- Deterministic and predictable (constructed, not hashed)
- Human-readable and debuggable
- Encodes routing information directly in the ID
- Interoperable across platforms and languages (Java, Python, Rust, Go, C#)
- Suitable for use in URLs, database keys, and log correlation

---

### 🔧 Format

A `messageId` follows the structure:

```text
<version>:<groupId>:<sequenceId>
```

| Component | Description | Example |
|-----------|-------------|---------|
| `version` | Protocol version number (currently `2`) | `2` |
| `groupId` | Logical group the message belongs to (e.g., user ID, tenant) | `user-123` |
| `sequenceId` | Unique identifier within the group (e.g., session, device) | `session-A` |

This ensures:
- 🛡️ **Transparency**: IDs are self-describing and carry routing metadata.
- 🔁 **Determinism**: The same `(groupId, sequenceId)` always produces the same `messageId`.
- 💡 **Debuggability**: IDs are human-readable, unlike opaque hashes.
- 🧩 **Cross-language compatibility**: Simple string concatenation — no cryptographic dependency.

---

### 📌 Construction

```text
version:    2
groupId:    "user-123"
sequenceId: "session-A"
messageId:  "2:user-123:session-A"
```

No hashing or encoding is required. Implementations simply join the three components with `:` as separator.

### 🎯 Example

For `groupId = "user-123"` and `sequenceId = "session-A"`:

```text
2:user-123:session-A
```

### 🌐 Language Implementations

All implementations produce the same output for the same inputs.

**Java**
```java
public class MessageIdGenerator {
    private static final int PROTOCOL_VERSION = 2;

    public static String generateId(String groupId, String sequenceId) {
        return PROTOCOL_VERSION + ":" + groupId + ":" + sequenceId;
    }
}
```

**C#**
```csharp
public static class MessageIdGenerator
{
    private const int ProtocolVersion = 2;

    public static string GenerateId(string groupId, string sequenceId)
        => $"{ProtocolVersion}:{groupId}:{sequenceId}";
}
```

**Python**
```python
PROTOCOL_VERSION = 2

def generate_message_id(group_id: str, sequence_id: str) -> str:
    return f"{PROTOCOL_VERSION}:{group_id}:{sequence_id}"
```

**Go**
```go
import "fmt"

const protocolVersion = 2

func GenerateMessageID(groupID, sequenceID string) string {
    return fmt.Sprintf("%d:%s:%s", protocolVersion, groupID, sequenceID)
}
```

**Rust**
```rust
const PROTOCOL_VERSION: u32 = 2;

pub fn generate_message_id(group_id: &str, sequence_id: &str) -> String {
    format!("{}:{}:{}", PROTOCOL_VERSION, group_id, sequence_id)
}
```

## 🚫 Token Revocation (v3.0.2 — Signed Tombstones)

As of v3.0.2, token revocation uses **signed tombstones** instead of empty messages. This prevents an attacker with broker access from forging or replaying revocation events.

### Tombstone Structure

A revocation tombstone is a broker event with the following properties:

- **key**: The `messageId` of the token to revoke (format: `<version>:<groupId>:<sequenceId>`)
- **message**: A signed tombstone payload containing:

| Field | Type | Description |
|-------|------|-------------|
| `signerId` | `string` | Identifier of the service issuing the revocation |
| `tombstoneSig` | `string` (Base64) | RSA signature over the canonical tombstone data, signed with the service's long-term private key |
| `timestamp` | `long` (epoch ms) | Timestamp of the revocation event |

### Conflict Resolution

When multiple tombstones target the same `messageId`, the **latest-timestamp-wins** rule applies:
- The tombstone with the highest `timestamp` value takes precedence.
- Replayed tombstones with older timestamps are silently discarded.

### Verification

All receiving sites **must**:
1. Verify `tombstoneSig` against the `signerId`'s known public key (resolved via `TrustAnchor`).
2. Reject tombstones with invalid or missing signatures.
3. Apply the latest-timestamp-wins rule when processing concurrent revocations.
4. Consider the targeted token as invalid only after successful tombstone verification.

### Example (Java)

```java
// Revoke a specific session — produces a signed tombstone
sv.revoke("user-123", "session-A");

// Revoke all sessions for a group
sv.revoke("user-123", null);
```

The `GenericSignerVerifier.revoke()` method automatically constructs and signs the tombstone using the service's long-term private key.
