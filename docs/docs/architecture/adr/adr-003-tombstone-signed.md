# ADR-003: Signed Revocation Tombstones

* **Status**: Superseded by [ADR-007](./adr-007-fence-tokens.md)
* **Date**: 2026-06-27

## Context

In Veridot V2, session revocation was accomplished by publishing an unsigned "tombstone" entry to the broker. This created a denial-of-service vector: any client or compromised node with broker write access could inject a tombstone and revoke active sessions for arbitrary users.

## Decision

We require a valid cryptographic signature on all revocation messages (tombstones):
- Revocation entries must contain a `tombstoneSig` generated by the issuer's long-term key.
- Verifier nodes validate the signature against the `TrustAnchor` before applying the revocation.
- Revocation entries are ordered using a latest-timestamp-wins strategy.

## Consequences

- Attackers can no longer inject fake revocations.
- Revocation propagation remains asynchronous but is now fully secured.
