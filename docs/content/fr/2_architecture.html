Architecture Système & Protocole V4

Veridot V4 est construit sur un modèle de réplication d'événements hautement optimisé. Le protocole garantit que tous les nœuds de vérification maintiennent la cohérence de l'état de sécurité sans goulot d'étranglement.

Format de l'Enveloppe Binaire

Toutes les transitions d'état (ex. CONFIG, KEY_EPOCH, LIVENESS) sont empaquetées dans une enveloppe binaire signée cryptographiquement. Son format a été conçu pour un parsing rapide et sans copie :

Offset (Octet) | Taille (Octets) | Nom du champ      | Description
---------------+-----------------+-------------------+-----------------------------------
0              | 1               | Magic Version     | Identifiant de protocole (doit être 0x04)
1              | 1               | Entry Type        | Type de payload (1-5)
2              | 1               | Flags             | Indicateurs de compression ou d'état
3              | 1               | Sig Alg           | Identifiant de l'algorithme de signature (ex. 0x01)
4-5            | 2               | Scope Length      | Longueur de la chaîne Scope
6-X            | Var             | Scope Value       | Chaîne UTF-8 (ex. "group:user-123")
X+1 - X+2      | 2               | Key Length        | Longueur de la sous-clé de session
X+3 - Y        | Var             | Key Value         | Chaîne UTF-8 (ex. "session-9942")
Y+1 - Y+8      | 8               | Version           | Numéro de version de séquence strictement croissant
Y+9 - Y+16     | 8               | Timestamp         | Date de publication en millisecondes
Y+17 - Y+18    | 2               | Issuer ID Length  | Longueur de l'identifiant de l'émetteur
Y+19 - Z       | Var             | Issuer ID Value   | Identifiant UTF-8 du signataire
Z+1 - Z+4      | 4               | Payload Length    | Longueur de la payload interne
Z+5 - W        | Var             | Payload Bytes     | Octets encodés de la payload
W+1 - W+2      | 2               | Signature Length  | Longueur de la signature cryptographique
W+3 - Fin      | Var             | Signature Bytes   | Signature calculée sur les octets canoniques

Types de Payloads du Protocole

Veridot définit 5 structures de payloads spécialisées encodées en format compact :

  • CONFIG (Type 0x01) : Définit les limites de capacité (ex. maximum 5 appareils actifs), la politique d'éviction (FIFO / REJECT) et les TTL par défaut.
  • KEY_EPOCH (Type 0x02) : Représente les clés éphémères de rotation. Contient la clé publique brute, l'intervalle de validité de rotation, et le jeton JWT optionnel (en mode indirect).
  • LIVENESS (Type 0x03) : Signal de vie des sessions actives. Statut configuré à ACTIVE (mises à jour régulières) ou REVOKED (déconnexion explicite).
  • FENCE (Type 0x04) : Jetons de barrière (FENCE) contenant un compteur monotone. Utilisé pour valider les écritures concurrentes et isoler les éditeurs.
  • SNAPSHOT_MARKER (Type 0x05) : Publié périodiquement lors des réconciliations. Définit le filigrane de base sur tous les serveurs.

Suivi de Version & Anti-Rollback

Pour prévenir les attaques par rejeu (où un attaquant réémet une ancienne enveloppe ACTIVE pour réactiver un jeton révoqué), Veridot applique **l'invariant de version monotone** :

Règle de Monotonicité

Pour chaque ressource unique (Scope + Type + Key), les validateurs conservent un filigrane local W(id). Lorsqu'une nouvelle enveloppe arrive avec la version V :

Si V ≤ W(id), l'enveloppe est immédiatement rejetée.

Si V > W(id), l'enveloppe est acceptée et W(id) ← V.

Cela garantit que l'état du système ne peut jamais reculer dans le temps, même si la base de données subit un retard de réplication.

Protocole de Fencing (§9.4)

Lorsque plusieurs nœuds tentent de modifier la capacité d'un groupe simultanément, Veridot utilise les **jetons FENCE** pour ordonner les écritures :
